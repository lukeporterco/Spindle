# SteelHook Architecture

This folder records broad SteelHook strategy and future SteelHook versioning direction.

SteelHook 0.1 Target pass documents live under `../minecraft-target/steelhook-0.1/` because they were implemented as Target-* passes. This folder should hold cross-version SteelHook strategy, future primitive planning, and documents that are not specific to a single Target pass sequence.

SteelHook 0.2 Target pass documents now begin under `../minecraft-target/steelhook-0.2/`. Target-23 selects the first approved primitive boundary as analysis-only planning input for later contract and transformer generalization work, Target-24 lifts that bounded primitive into reusable descriptors for Target-25 without transforming Minecraft classes, Target-25 proves the reusable method-entry transformer through offline-only class-byte rewriting while still leaving runtime classloading, hook installation, and dispatch disabled, Target-26 proves the gated runtime classloader path by defining the transformed Minecraft target class without invoking `Main.main`, launching Minecraft, installing hooks, invoking runtime dispatch, exposing public API, or claiming Java mod execution sandboxing, and Target-27 closes SteelHook 0.2 in the narrow capability-ladder sense by verifying the full report chain and handoff state.

SteelHook 0.2 therefore means one approved gated runtime classloader transformation path only. It does not mean arbitrary bytecode editing, Mixin replacement, public SteelHook APIs, Java mod execution sandboxing, StackMapTable rewriting, method-exit hooks, cancellable hooks, callsite redirects, return-value interception, field hooks, constructor hooks, multi-hook composition, or registry/command/resource/lifecycle implementation.

SteelHook 0.3 is now complete through Target-31. Its completed capabilities are a framed-method foundation for the existing method-entry static-dispatch primitive, a sibling offline-only `METHOD_EXIT_STATIC_DISPATCH` proof that inserts `invokestatic` immediately before supported normal return opcodes in controlled unframed fixtures, a generalized gated runtime proof that defines each approved primitive through an isolated runtime classloader session, and a deterministic completion handoff report `minecraft-steelhook-0-3-report.json`. SteelHook 0.3 still does not compose hooks into one class, install hooks, invoke Minecraft main, launch Minecraft, execute dispatchers, expose public APIs, claim Java mod execution sandboxing, add full StackMapTable recomputation, exceptional-exit support, or return-value interception.

SteelHook 0.4 has begun, and it remains incomplete after Target-34. Target-32 is the boundary-definition pass that approves only the planned internal primitive families `RETURN_VALUE_INTERCEPT`, `INVOKE_REDIRECT`, and `INVOKE_WRAP`, records their allowed fixture shapes, unsupported shapes, evidence requirements, and rejection taxonomy, and keeps them internal-only until later evidence exists. Target-33 proves only `RETURN_VALUE_INTERCEPT`, stays offline-only, supports only controlled single primitive and reference return fixtures, reports hashes and deterministic evidence instead of raw byte payloads, and still does not perform runtime classloading, class definition, hook installation, dispatcher execution, Minecraft execution, public API exposure, or Java mod execution sandbox claims. Target-34 proves only `INVOKE_REDIRECT` and `INVOKE_WRAP`, stays offline-only, supports only one controlled `invokestatic` callsite shape, requires strict owner, name, descriptor, and opcode matching, rewrites only the methodref operand, records wrapped delegate metadata for wrap without executing the wrapper, reports hashes and deterministic evidence instead of raw byte payloads, and still does not perform runtime classloading, class definition, hook installation, dispatcher execution, Minecraft execution, public API exposure, or Java mod execution sandbox claims. Targets 35 and 36 remain required before SteelHook 0.4 can be called complete.
